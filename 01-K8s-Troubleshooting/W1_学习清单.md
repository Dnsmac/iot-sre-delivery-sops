## 学习顺序（按天，每天 ≤3 个原子任务）

### D1（≈45min×2）：K8s 最小必要概念 — 对照你的集群

**学什么**

| 概念 | 只需到这个深度 | 和你环境的关系 |
|------|----------------|----------------|
| Pod | 最小运行单元，会重启、会换 IP | 所有组件都跑在 Pod 里 |
| Deployment | 无状态多副本 | **iot-server** 典型 |
| StatefulSet | 稳定网络标识 + 常配 PVC | **Pulsar/MySQL/Redis/ES** 常见 |
| Service ClusterIP | 集群内访问 | 服务互调 |
| Service LoadBalancer | 对外要云/MetalLB 给 IP | **MetalLB 存在的理由** |
| Namespace | 隔离与配额 | 你们是否按中间件分 ns |
| 探针 liveness/readiness | 决定杀不杀 Pod、进不进 Service | 适配时误杀过？ |
| requests/limits | CPU/内存上限，OOM 来源 | Java/ES 最容易踩 |

**怎么学（只读现网）**

```bash
kubectl get ns
kubectl get deploy,sts,svc -A | findstr /i "pulsar mysql redis es nacos minio iot metallb"
# 任选一个 Pod（把 <pod> <ns> 换成真实值）
kubectl get pod <pod> -n <ns> -o wide
kubectl describe pod <pod> -n <ns>
kubectl describe svc <svc> -n <ns>
```

**D1 自检（口答或写 3 行笔记）**

- [ ] Deployment 和 StatefulSet 各举你环境里的 **1 个组件名**
- [ ] 说出 `kubectl describe pod` 里 **Events** 段是干什么的
- [ ] 找到一个带 `limits` 的容器，说出 memory limit 数值

---

### D2：网络与暴露 — MetalLB + Service

**学什么**

| 概念 | 深度 |
|------|------|
| ClusterIP / NodePort / LoadBalancer | 三种 Service 区别，你们对外用哪种 |
| MetalLB | 给 LoadBalancer Service 分 **内网 VIP** 的流程（L2 即可） |
| Ingress vs LB | Ingress 是 HTTP 路由规则；MQTT/TCP 常不走 Ingress |
| DNS | `服务名.namespace.svc` 集群内解析 |

**怎么学**

```bash
kubectl get svc -A | findstr LoadBalancer
kubectl describe svc <带EXTERNAL-IP的svc> -n <ns>
kubectl get pods -n metallb-system
kubectl logs -n metallb-system -l app=metallb --tail=50
```

**D2 自检**

- [ ] 画一条：`外部客户端 → MetalLB IP:端口 → Service → Pod:容器端口`
- [ ] 说出 iot-server 对外是 **LB / NodePort / Ingress** 哪一种（去现网确认）
- [ ] MetalLB 的 IP 池大概在哪个网段（CIDR 或描述即可）

---

### D3：有状态 — Pulsar 集群 + 消息在链路中的位置

**学什么**

| 概念 | 深度 |
|------|------|
| Pulsar 架构 | Broker + BookKeeper（存储）+（可能）ZK；不必背安装 |
| K8s 上 Pulsar | 多为 StatefulSet；集群名、存储卷 |
| 生产问题类型 | 积压、订阅滞后、Bookie 磁盘满、命名空间配额 |
| 与 iot-server | 设备上报/下发 **是否经 Pulsar**（去配置确认） |

**怎么学**

```bash
kubectl get sts,svc,pvc -n <pulsar的ns>
kubectl get pods -n <pulsar的ns> -o wide
# 若容器内有 pulsar-admin（没有则跳过，只记「应用层命令名」）
kubectl exec -it <broker-pod> -n <ns> -- pulsar-admin brokers list 2>nul
```

**扩展（可选 15min）**：Pulsar 官方文档只读「Concepts → Architecture」一章。

**D3 自检**

- [ ] Pulsar 在 K8s 里是 Deploy 还是 STS
- [ ] BookKeeper 数据存在 **PVC 还是 hostPath**（describe pvc 看）
- [ ] 说出一个你们适配时和 Pulsar 有关的 **关键词**（端口/鉴权/存储/副本）

---

### D4：有状态 — MySQL 三主 + 达梦（国产化）

**学什么**

| 概念 | 深度 |
|------|------|
| MySQL 三主 | 是 **三主写入** 还是 **一主多从/ MGR**（必须去现网确认，名词不能猜） |
| K8s 部署 | STS + Headless Service 常见；读写分离是否在应用侧 |
| 达梦 | 集群内还是集群外；与 MySQL 业务分工（哪些表/库） |
| 风险 | 主切换、连接池爆满、慢查询拖垮 iot-server |

**怎么学**

```bash
kubectl get sts,svc,pvc -n <mysql的ns>
kubectl describe sts <mysql-sts名> -n <ns>
# 应用配置里搜 jdbc（在 Nacos 或 iot-server 的 configmap，只读）
kubectl get cm -n <ns> | findstr /i "iot nacos mysql dameng"
```

**D4 自检**

- [ ] 画出：iot-server → **连 MySQL 还是达梦还是两个都有**
- [ ] MySQL 三主的 **真实拓扑名词**（自己查完写一句，不要抄网上的「三主」）
- [ ] 达梦在清单里补全：单点/主备/集群

### D5：Redis 集群 + ES 集群

**学什么**

| 组件 | 深度 |
|------|------|
| Redis 集群 | 分片还是哨兵；K8s 用 STS 还是 Operator；iot-server 用的 **集群模式还是单机地址** |
| ES 集群 | 节点角色（master/data）；堆内存 limits；索引谁写（日志还是业务） |
| 共性 | 内存打满、磁盘 watermark、网络分区时的现象 |

**怎么学**

```bash
kubectl get sts,deploy,svc -n <redis的ns>
kubectl get sts,svc -n <es的ns>
kubectl describe pod <es-pod> -n <ns> | findstr -i "Limits memory"
```

**D5 自检**

- [ ] Redis：集群节点数 + Service 名
- [ ] ES：memory limit 大概多少；用途 **日志 / 检索 / 两者**
- [ ] 各说一个「满了会怎样」的现象（CPU/内存/磁盘 任一种）

---

### D6：Nacos + MinIO + iot-server 启动链

**学什么**

| 组件 | 深度 |
|------|------|
| Nacos | 注册中心 vs 配置中心；iot-server 启动先拉哪类配置 |
| MinIO | bucket 概念；存什么（固件/图片/导出包） |
| iot-server | 依赖顺序：Nacos → DB → Redis → Pulsar？（以你项目为准） |
| 探针 | readiness 失败时 Service 会不会导流量 |

**怎么学**

```bash
kubectl get deploy iot-server -n <ns> -o yaml > %TEMP%\iot-deploy.yaml
# 打开文件只看：env、volumeMount、livenessProbe、readinessProbe、resources
kubectl get deploy,svc -n <nacos的ns>
kubectl get deploy,svc -n <minio的ns>
```

**D6 自检**

- [ ] 列出 iot-server 的 **3 个环境依赖**（主机名级别）
- [ ] Nacos 配置变更有无导致过 **全服务重启**（回忆即可）
- [ ] MinIO 对外是 API 端口多少（或 Console 与 API 是否分开）
