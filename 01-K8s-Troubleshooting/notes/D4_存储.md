## 0. 为什么需要这套抽象？

| 问题 | 解决 |
|------|------|
| Pod 删了数据没了 | PVC 独立于 Pod 生命周期 |
| 开发者不该关心 NFS/云盘细节 | PV 抽象底层；PVC 只声明「要多大、什么模式」 |
| 手工建 PV 太慢 | **StorageClass** 动态供给 |

**一句话**：**StorageClass 定类型 → PVC 提需求 → PV 给资源 → Pod 挂载 PVC**。

---

## 1. 三大概念

### 1.1 PersistentVolume（PV）— 提供者

| 项 | 说明 |
|----|------|
| 角色 | **集群级**存储资源（不属于某个 Namespace） |
| 谁管 | 管理员或 **Provisioner 动态创建** |
| 抽象 | NFS、iSCSI、云盘、Ceph、Local 等 |
| 生命周期 | **独立于 Pod**；Pod 删了 PV 数据仍在（看回收策略） |

**关键 spec**：

| 属性 | 说明 |
|------|------|
| `capacity` | 如 `10Gi` |
| `accessModes` | RWO / ROX / RWX |
| `persistentVolumeReclaimPolicy` | Retain / Delete / Recycle（已废弃） |
| `storageClassName` | 与 PVC、StorageClass 对齐 |
| `status` | Pending → Bound → Released → Failed |

### 1.2 PersistentVolumeClaim（PVC）— 消费者

| 项 | 说明 |
|----|------|
| 角色 | **Namespace 级**存储申请 |
| 类比 | Pod 消耗 Node CPU/内存；**PVC 消耗 PV 容量** |
| 绑定 | 控制器找 **满足条件** 的 PV，**1:1 绑定** |

**用户声明**：容量、accessModes、storageClassName、可选 selector。

### 1.3 StorageClass — 类型模板

| 项 | 说明 |
|----|------|
| 角色 | 描述存储 **类别**（SSD/HDD/NFS/本地盘） |
| 核心 | **动态供给**：PVC 指定 SC → 自动建 PV + 底层盘 |
| 字段 | `provisioner`、`parameters`、`reclaimPolicy`、`allowVolumeExpansion` |

**常见 Provisioner**：

| 类型 | Provisioner 示例 | 底层 |
|------|------------------|------|
| 云盘 | `kubernetes.io/aws-ebs` 等 | EBS/云盘 |
| NFS | NFS 类 provisioner | NFS/NAS |
| 本地 | `kubernetes.io/no-provisioner` + Local PV | 节点本地盘 |
| Ceph/Rook | `cephfs.rook.io` 等 | Ceph RBD |

## 2. 访问模式（面试必背）

| 模式 | 缩写 | 含义 | 常见场景 |
|------|------|------|----------|
| ReadWriteOnce | **RWO** | **单节点**读写 | 云盘、Most DB |
| ReadOnlyMany | ROX | 多节点只读 | 配置分发 |
| ReadWriteMany | **RWX** | 多节点读写 | **NFS**、共享文件 |

**绑定规则**：PVC 请求的 modes 必须是 PV modes 的 **子集**（如 PVC 要 RWO，PV 是 RWX ✅）。

---

## 3. 三者关系与供给方式

```text
StorageClass（模板）
    ↓ 动态供给
PV（资源） ←──1:1──→ PVC（申请） ←──挂载── Pod
```

### 3.1 静态供给

1. 管理员 **手动创建 PV**（带真实 NFS/云盘信息）
2. 用户创建 **PVC**
3. 控制器 **匹配并绑定**

### 3.2 动态供给（生产常见）

1. 管理员创建 **StorageClass**
2. 用户创建 PVC 并写 `storageClassName`
3. **Provisioner** 自动创建底层存储 + **PV**，再绑定 PVC  
4. **无需事先手工建 PV**

---

## 4. PVC 绑定 PV 的规则

| 条件 | 要求 |
|------|------|
| 容量 | PVC 请求 ≤ PV 容量（如 PVC 5Gi，PV 10Gi ✅） |
| 访问模式 | PVC modes ⊆ PV modes |
| StorageClass | 名称 **一致**（或都为空） |
| Selector | PVC 的 label selector 需匹配 PV labels |
| 关系 | **1:1**，一个 PV 同时只能绑一个 PVC |

**Pending 常见原因**：无合适 PV、SC 不一致、容量/模式不匹配、PV 已 Bound。

---

## 5. 生命周期

| 阶段 | 说明 |
|------|------|
| **Provisioning** | 创建 PV（静态或动态） |
| **Binding** | PVC 绑定到 PV |
| **Using** | Pod `volumeMounts` 使用 PVC |
| **Releasing** | Pod 删、PVC 删 → PV 变 **Released** |
| **Reclaiming** | 按策略处理数据 |

### 5.1 回收策略

| 策略 | PVC 删除后 | 数据 | 场景 |
|------|------------|------|------|
| **Retain** | PV 保留，状态 Released | **保留** | 生产库、防误删 |
| **Delete** | PV 删除 | 底层卷常 **一并删** | 动态云盘、临时数据 |
| Recycle | 已废弃 | 清空 | 用动态+Delete 代替 |

Released 的 PV **不会自动**给新 PVC 用，需管理员清理或手工复用。

## 6. Pod 如何使用存储

```yaml
# STS 常用 volumeClaimTemplates 自动为每个 Pod 建 PVC
volumeMounts:
  - name: data
    mountPath: /var/lib/mysql
volumes:
  - name: data
    persistentVolumeClaim:
      claimName: mysql-pvc
```

- **Deployment** 也可挂 PVC，但多副本 **RWO 云盘** 只能挂 **一个节点**（多副本要用 RWX 或每 Pod 独立 PVC）。
- **StatefulSet** + `volumeClaimTemplates`：Pod-0 → pvc-0，Pod-1 → pvc-1（见 D1 §3）。

## 7. 职责分离（管理员 vs 开发者）

| 管理员 | 开发者 |
|--------|--------|
| 建 PV、定义 StorageClass、管 NFS/存储阵列 | 建 PVC、在 Pod/STS 里挂载 |
| 关心「有什么存储」 | 关心「要多大、RWO 还是 RWX」 |

---

## 8. 结合现项目（有状态组件）

| 组件 | 存储相关 | 你要确认的 |
|------|----------|------------|
| Pulsar / BookKeeper | 常有 **PVC** | `kubectl get pvc -n <pulsar-ns>` |
| MySQL 三主 | **STS + volumeClaimTemplates** | `kubectl get sts -o yaml` 看 volumeClaimTemplates |
| Redis / ES | 常有 PVC | 容量、是否 RWO |
| MinIO | 对象存储，可能 PVC 或本地盘 | 现网为准 |
| iot-server | 多为无状态 Deploy，**可无 PVC** | 以 describe 为准 |

**排查命令**：

```bash
kubectl get storageclass
kubectl get pv
kubectl get pvc -A
kubectl get pvc -n <ns>
kubectl describe pvc <name> -n <ns>
```

## 9. 静态 NFS 流程速记（实战）

```text
NFS 服务端导出目录 → 管理员建 PV (nfs.server/path)
→ 开发者建 PVC (匹配 SC、RWX、容量)
→ Pod/STS 挂载 PVC
→ kubectl get pv/pvc 看 Bound
```

## 11. 面试速记

| 问题 | 要点 |
|------|------|
| PV vs PVC？ | 提供 vs 申请；集群级 vs NS 级；职责分离 |
| StorageClass？ | 动态供给 + 存储分级 |
| 为何分离？ | 解耦底层细节、生命周期独立 |
| Pending？ | 容量/模式/SC/无空闲 PV |
| Retain vs Delete？ | 保留数据 vs 删卷删数据 |
| STS 存储？ | volumeClaimTemplates，每 Pod 独立 PVC |
