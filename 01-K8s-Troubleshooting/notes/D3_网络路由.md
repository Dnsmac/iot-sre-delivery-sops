## 0. 本文范围与模块关系

| 组件 | 层级 | 一句话 | 与现网 |
|------|------|--------|--------|
| **CNI** | L2/L3 | Pod 之间怎么通 | 待确认集群用 Calico/Flannel 等 |
| **Service** | L4 | 稳定入口 + 负载均衡 | iot-web **ClusterIP** + **LB VIP** |
| **MetalLB** | L4 | 裸机给 LB 分 VIP | 见 [`D2_MetalLB.md`](D2_MetalLB.md) |
| **Ingress** | L7 | HTTP/HTTPS 域名路径路由 | IoT **1883** 通常不走 Ingress |
| **NetworkPolicy** | L3/L4 | Pod 级防火墙 | 依赖 CNI 是否支持 |

```text
外部 → (Ingress L7) → Service VIP L4 → kube-proxy → Pod IP (CNI)
裸机外部 → MetalLB VIP → kube-proxy → Pod IP (CNI)    ← 你们 iot-web 主路径
```

## 2. Service：服务发现与负载均衡

### 2.1 为什么需要 Service？

- Pod 重启/扩缩容 → **IP 会变**，直接访问 Pod IP 不可靠。
- Service 为一组 Pod（**Label Selector**）提供 **稳定的服务名 + 端口**（ClusterIP 或 DNS）。

### 2.2 Service 不转发流量 — kube-proxy 才转发

| 步骤 | 说明 |
|------|------|
| 监听 | 每节点 **kube-proxy** 监听 Service、Endpoints 变化 |
| 写规则 | 按 **iptables** 或 **IPVS** 生成转发规则 |
| 转发 | 访问 Service VIP → 规则转到后端 **Pod IP** |

```text
Client → DNS (svc.ns.svc.cluster.local)
      → Service VIP（虚拟 IP，不在网卡上）
      → kube-proxy (iptables/ipvs)
      → Endpoints（Pod IP 列表）
      → Pod（CNI 负责到 Pod 的网络）
```

### 2.3 四种类型 + Headless

| 类型 | 访问范围 | 原理 | 适用场景 |
|------|----------|------|----------|
| **ClusterIP** | 集群内 | 虚拟 VIP，kube-proxy 负载 | 默认；微服务互调、数据库内网 |
| **NodePort** | 集群外也可 | 每节点开 **30000–32767** 端口 | 调试；LB 类型会 **附带** NodePort |
| **LoadBalancer** | 集群外 | 云 LB 或 **MetalLB** 分配 VIP | 生产对外；**iot-web .190** |
| **ExternalName** | DNS 映射 | 无 VIP，CNAME 到集群外域名 | 集群内访问外部库 |
| **Headless** | 集群内 | `clusterIP: None`，DNS 解析到 **各 Pod IP** | **StatefulSet**、Pulsar/MySQL 等 |

### 2.4 kube-proxy：iptables vs IPVS

| 模式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **iptables** | 规则链匹配 | 简单成熟 | 规则多时性能降 |
| **IPVS** | 哈希表转发 | 大规模高性能 | 需内核 `ip_vs`；+ MetalLB L2 时可能要 **strictARP**（见 D2 §7.2） |

### 2.5 Endpoints

- **Endpoints** = Service 与 Pod 的桥梁；`kubectl get endpoints` 看实时后端 IP。
- 无 Pod 匹配 selector → Endpoints **空** → VIP 通但无后端（排错见 D2 §8.6）。
- **手动 Endpoints**：可把集群外 IP（如物理机 MySQL）挂到 ClusterIP Service 上。

### 2.6 现网要点（与 D2 合并）

| 字段 | `meta/iot-web` 示例 |
|------|---------------------|
| ClusterIP | `10.233.40.193`（集群内） |
| EXTERNAL-IP | `192.168.27.190`（MetalLB L2） |
| NodePort | `80:30673`、`1883:30189`（附带，设备用 VIP 不用 NodePort） |
| 后端 | 3 个 Endpoint |

## 3. 裸机 LoadBalancer：MetalLB（详见 D2）

- 云上 LoadBalancer → 云厂商 API 分配 CLB/ELB。
- **裸机/IDC** → **MetalLB** 写 `status.loadBalancer.ingress.ip`，Speaker L2/BGP 宣告。

## 4. Ingress：集群统一入口（L7）

### 4.1 为什么需要 Ingress？

| 对比 | Service (NodePort/LB) | Ingress |
|------|------------------------|---------|
| 层级 | **L4** TCP/UDP | **L7** HTTP/HTTPS |
| 域名/路径 | 不能按 Host/Path 分流 | **虚拟主机、路径路由** |
| TLS | 应用或 LB 各自配 | **Ingress 统一 TLS 终结** |
| 成本 | 每服务一个 LB/端口 | **一个入口**代理多 HTTP 服务 |

### 4.2 两个组成部分

| 部分 | 作用 |
|------|------|
| **Ingress 资源** | YAML 规则：Host、Path、backend Service |
| **Ingress Controller** | 真正执行（如 **ingress-nginx**）；监听 API，改 Nginx/Envoy 配置 |

**面试句**：Ingress 只是规则；**Ingress Controller** 才是干活的。

### 4.3 流量路径

```text
外部 HTTP(S) → Ingress Controller → Service ClusterIP → kube-proxy → Pod
```

### 4.4 常见 Controller

| 控制器 | 特点 |
|--------|------|
| ingress-nginx | 通用、社区大 |
| Traefik | CRD、热重载 |
| Kong | 插件、治理 |
| 云 ALB/SLB | 托管、云上生产 |

### 4.5 IoT 与 Ingress

- **MQTT 1883** 是 TCP，一般 **不用标准 HTTP Ingress**（除非 TCP Ingress / 专用网关）。
- 你们 **iot-web** 用 **LoadBalancer + VIP:1883** 合理（D2 §7.1）。

## 5. NetworkPolicy：网络访问控制

### 5.1 作用

- Pod 级 **防火墙**：控制 **谁可以访问我（Ingress）**、**我可以访问谁（Egress）**。
- 类似云 **安全组**，按 **标签、命名空间** 选 Pod。

### 5.2 默认行为 vs 策略语义

| 项 | 说明 |
|----|------|
| **默认** | K8s **全开放**，任意 Pod 可互访（flat 网络） |
| 启用 NP 后 | 对选中 Pod 常表现为 **白名单**：只放行规则里的流量 |
| 类型 | `Ingress`（入站）、`Egress`（出站） |

### 5.3 实现依赖 CNI

| CNI | NetworkPolicy |
|-----|---------------|
| **Calico** | ✅ Felix 实现 |
| **Cilium** | ✅ eBPF |
| **Flannel** | ❌ 默认不支持 |
| **Canal** | Flannel 网络 + Calico 策略 |

**面试句**：NP 是 API；**能不能生效看 CNI**。

## 6. CNI：Pod 网络基础设施

### 6.1 CNI 职责

- 给 Pod 分配 **IP**、配置 **网络命名空间**、保证 **跨节点 Pod 可通信**（扁平网络，通常 Pod 间不经 NAT）。
- 同 Pod 内容器：**loopback**；跨 Pod：veth → bridge/路由 → CNI 后端。

```text
Pod eth0 → veth → 节点网络 → (同节点或跨节点) → 对端 Pod
```

### 6.2 常见方案对比

| 插件 | 模式 | 性能 | NetworkPolicy | 场景 |
|------|------|------|---------------|------|
| **Flannel** | Overlay VxLAN | 中 | ❌ | 学习、小集群 |
| **Calico** | BGP 或 IPIP | 高 | ✅ | **生产常用** |
| **Cilium** | eBPF | 很高 | ✅ | 高性能、可观测 |
| **Canal** | Flannel + Calico 策略 | 中 | ✅ | 要简单 overlay + 策略 |

### 6.3 Flannel vs Calico（面试高频）

| 维度 | Flannel | Calico |
|------|---------|--------|
| 跨节点 | VxLAN **封装**，有开销 | **BGP 路由**或 IPIP，可走原生路由 |
| NetworkPolicy | ❌ | ✅ |
| 复杂度 | 低 | 较高（BGP 要网络配合） |
| 大规模 | 一般 | 较好 |

### 6.4 Calico 架构（了解）

- **Felix**：节点上执行路由和策略。
- **BIRD**：BGP 宣告 Pod 网段（与 **MetalLB BGP** 不是一回事）。
- 配置存储常对接 **etcd** 或 K8s API。

### 6.5 跨主机 Pod 通信两种思路

| 方式 | 代表 | 特点 |
|------|------|------|
| **Overlay** | Flannel VxLAN | 封装，配置简单，有 10～20% 量级开销（视流量） |
| **路由** | Calico BGP | 性能好，要交换机/路由器支持 BGP |

## 7. 完整南北向流量（HTTP vs IoT）

### 7.1 典型 HTTP（走 Ingress）

```text
DNS → 公网 IP → Ingress Controller (L7)
    → Service ClusterIP (L4) → kube-proxy → Pod IP (CNI)
```

### 7.2 你们 IoT（走 MetalLB LB，D2）

```text
设备 → 192.168.27.190:80 / :1883 (L4 VIP, MetalLB L2)
    → node6 入口 → kube-proxy → iot-web Endpoints → Pod (CNI)
```

### 7.3 集群内调用

```text
Pod A → iot-web.meta.svc:port → ClusterIP 10.233.40.193 → kube-proxy → Pod B
```
