## 0. 架构二分

| 平面 | 职责 | 主要组件 |
|------|------|----------|
| **控制平面** | 集群「大脑」：决策、调度、状态存储 | apiserver、etcd、scheduler、controller-manager |
| **数据平面** | 工作节点：跑 Pod、网络转发、容器 | kubelet、kube-proxy、容器运行时 |

```text
用户/kubectl → apiserver ↔ etcd
                ↓ watch
         scheduler / controller-manager
                ↓
         kubelet (各节点) → CRI → 容器
         kube-proxy → Service 转发（见 D3）
```

## 1. 控制平面

### 1.1 kube-apiserver — 唯一入口

| 项 | 说明 |
|----|------|
| 本质 | **REST API**；所有组件通过它读写集群状态 |
| 安全 | 认证、鉴权、准入控制 |
| 持久化 | 对象写入 **etcd**（不直连 etcd） |
| 机制 | **Watch**：资源变化推送给 scheduler、kubelet 等 |
| 高可用 | 多实例 + 前端 LB（你们裸机也有 VIP 访问 API 的场景） |

**面试句：apiserver 挂了 → 已运行 Pod **仍可通信**（CNI/kube-proxy）；**不能**调度新 Pod、不能 kubectl 改资源，直到恢复。

### 1.2 etcd — 集群记忆

| 项 | 说明 |
|----|------|
| 类型 | 分布式 **KV**，Raft 共识 |
| 存什么 | 所有 K8s 对象（Pod、Service、Deployment、Secret…） |
| 路径 | 如 `/registry/pods/...` |
| 原则 | **只通过 apiserver 访问**；不应对外直连 |
| 高可用 | **奇数**节点（3/5） |

```text
kubectl → apiserver → etcd 读写 → 返回
```

### 1.3 kube-scheduler — 调度

**流程**：

1. 监听 **未绑定 node** 的 Pod  
2. **预选 Predicate**：筛掉资源不够、污点不容忍、端口冲突、亲和不满足的节点  
3. **优选 Priority**：打分选最优（负载、亲和、数据局部性等）  
4. 写回 `pod.spec.nodeName`

**调度失败常见原因**：CPU/内存不足、污点、nodeSelector、亲和/拓扑约束。

### 1.4 kube-controller-manager — 控制循环集合

| 控制器 | 职责 | 关联你已学 |
|--------|------|------------|
| Deployment | 管 RS、滚动更新 | D1 |
| ReplicaSet | 维持副本数 | D1 |
| Node | 节点失联 → NotReady、驱逐 | 运维 |
| Endpoint | Service ↔ Pod 列表 | D3 |
| Job/CronJob | 批处理/定时 | D1 |
| PV | 绑定、回收等 | D4 |

**原理**：期望状态（Spec）vs 实际（Status）→ 有差异则调谐（建/删/改 Pod 等）。与 D1 §1 控制循环一致。

### 1.5 cloud-controller-manager（了解）

- **云上**才有：对接 ELB、云盘、节点元数据等。
- **裸机/IDC**：常无或精简；你们 **MetalLB** 代替云 LB（D2）。

## 2. 数据平面（每个 Node）

### 2.1 kubelet — 节点代理

| 功能 | 说明 |
|------|------|
| 监听 | 本节点应跑的 Pod（apiserver 分配后） |
| 执行 | 通过 **CRI** 拉镜像、启停容器 |
| 探针 | 执行 liveness/readiness/startup |
| 上报 | 节点与 Pod 状态、资源用量 |
| 协作 | 挂 **PVC**（D4）、调 **CNI**（D3） |

**vs kube-proxy**：kubelet 管 **Pod 生命周期**；kube-proxy 管 **Service 流量规则**。

### 2.2 kube-proxy — Service 转发

| 项 | 说明 |
|----|------|
| 监听 | Service、Endpoints 变化 |
| 实现 | **iptables** 或 **IPVS** 写规则 |
| 效果 | 访问 ClusterIP → 转到后端 Pod IP |

详见过 [`D3_网络路由.md`](D3_网络路由.md) §2；与 MetalLB 分工：MetalLB 管 **进集群的 VIP**，kube-proxy 管 **集群内 Service**。

### 2.3 容器运行时（CRI）

| 运行时 | 说明 |
|--------|------|
| **containerd** | 现网最常见，K8s 推荐 |
| **CRI-O** | Red Hat 系 |
| Docker | 早期经 dockershim，**已废弃**；Docker 底层也常是 containerd |

```text
kubelet → CRI 接口 → containerd/CRI-O → 容器进程
```

## 3. 组件协作：创建一个 Pod

```text
1. kubectl apply → apiserver（认证/准入）→ etcd 保存
2. scheduler 绑定 nodeName → apiserver
3. 目标节点 kubelet watch 到 Pod → CRI 拉镜像起容器
4. kubelet 上报状态 → apiserver → etcd
5. Endpoint Controller 更新 Endpoints → kube-proxy 更新规则
6. （若 type=LB）MetalLB Controller 分 VIP → Speaker 宣告（D2）
```

---

## 4. 控制平面 vs 数据平面（面试）

| 问题 | 要点 |
|------|------|
| 边界 | 控制平面 **不跑业务 Pod**（除特例）；业务在 Node 上 |
| apiserver 挂 | 存量 Pod 跑；**无新调度、无 API 变更** |
| etcd 挂 | apiserver 无法持久化 → **集群不可用** |
| 单节点挂 | 该节点 Pod 受影响；控制平面 HA 可扛部分节点故障 |

### 8.1 一张表记住谁是谁

| 组件 | 平面 | 一句话 |
|------|------|--------|
| kube-apiserver | 控制 | **唯一 API 入口**；写 etcd；Watch |
| etcd | 控制 | 存全部对象；**只经 apiserver 访问** |
| kube-scheduler | 控制 | 给未调度 Pod **选节点**（预选+优选） |
| kube-controller-manager | 控制 | 多控制器；**Spec vs Status 调谐** |
| cloud-controller-manager | 控制 | **云上** ELB/云盘；裸机常无，用 MetalLB |
| kubelet | 数据 | 本节点 **起停容器**、探针、上报 |
| kube-proxy | 数据 | **Service** 转发（iptables/ipvs） |
| containerd/CRI-O | 数据 | **真正跑容器**（经 CRI） |

### 8.2 高频问答（口述用）

| 问题 | 要点 |
|------|------|
| 集群有哪些核心组件？ | 上表；控制平面 4 + 节点 kubelet/kube-proxy/运行时 |
| apiserver 作用？ | REST API、认证鉴权、唯一写 etcd 的入口 |
| **apiserver 挂了？** | **已运行 Pod 还能互访**；不能调度/不能 kubectl 改资源 |
| etcd 存什么？能直连吗？ | 所有 K8s 对象；**不能直连**，必须走 apiserver |
| scheduler 做什么？ | 预选（能跑吗）→ 优选（哪个最好）→ 写 nodeName |
| Pod 调度失败？ | 资源不足、污点、亲和、selector、拓扑约束 |
| controller-manager？ | Deployment/RS/Node/Endpoint… **控制循环** |
| kubelet vs kube-proxy？ | kubelet **管 Pod**；kube-proxy **管 Service 流量** |
| kube-proxy 两种模式？ | **iptables**（规则多变慢）/ **IPVS**（大规模更好） |
| 容器运行时？ | **containerd** 主流；Docker+dockershim **已废弃** |
| 从 apply 到 Running？ | apiserver→etcd→scheduler→kubelet→上报→Endpoint→kube-proxy（§3） |
| 控制平面 vs 数据平面？ | 大脑决策 vs 节点执行；业务 Pod 在数据平面 |
| etcd 为何奇数节点？ | Raft **多数派** 高可用 |
| Node NotReady？ | Node 控制器标记；久了 **驱逐** Pod |
| 业务流量经过 apiserver 吗？ | **不经过**；走 CNI + kube-proxy |
| 裸机 LB 谁做？ | **MetalLB**（数据平面 Speaker），不是 cloud-controller |

### 8.3 初级 / 中级 / 高级（原汇总表）

| 级别 | 典型问题 |
|------|----------|
| 初级 | 有哪些组件？apiserver 干啥？ |
| 中级 | etcd 与 apiserver 关系？scheduler 两阶段？kubelet 和 kube-proxy 区别？ |
| 高级 | apiserver 挂了 Pod 怎样？控制/数据平面边界？ |

### 8.4 和 D1～D4 的口诀

| 模块 | 底层里对应谁 |
|------|----------------|
| D1 Workload | **controller-manager** 里的 Deploy/STS 控制器 |
| D3 网络 | **kube-proxy** + Endpoint 控制器 |
| D2 MetalLB | **Speaker DaemonSet**（数据平面）；Controller 像 mini controller |
| D4 存储 | **PV 控制器**（在 controller-manager 内） |
