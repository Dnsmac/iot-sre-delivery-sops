## 一、核心概念

### 1. 三者角色

| 概念 | 角色 |
|------|------|
| **PV** | 集群存储 **提供者**（底层 NFS/云盘等的抽象） |
| **PVC** | **消费者**（用户按 NS 申请） |
| **StorageClass** | **模板/类型**（驱动动态建 PV） |

### 2. 为何分离？

- **职责分离**：管理员管底层；开发者只声明容量和模式。
- **抽象**：开发者不关心 NFS 还是 Ceph。
- **生命周期**：Pod 删了 PVC 可保留；数据策略由 reclaimPolicy 控制。

### 3. PV 作用域

**集群级**（无 namespace）；PVC 是 **Namespace 级**。

### 4. StorageClass 价值

- **动态供给**：PVC 指定 SC → 自动创建 PV + 底层卷。
- **存储分级**：gold/silver、SSD/HDD 等。
- **Provisioner 插件化** 对接不同存储系统。

### 5. 静态 vs 动态

| | 静态 | 动态 |
|---|------|------|
| PV | 管理员先建 | SC + Provisioner **自动建** |
| 流程 | PV → PVC 绑定 | PVC → 自动 PV → 绑定 |
| 场景 | 固定已知容量 | 生产主流、多租户 |

### 6. 几对几？

**1:1**。一个 PV 同时只能绑定一个 PVC。

## 二、访问模式与绑定

### 7. RWO / ROX / RWX

| 缩写 | 含义 | 场景 |
|------|------|------|
| RWO | 单节点读写 | 云盘、单副本 DB |
| ROX | 多节点只读 | 共享只读配置 |
| RWX | 多节点读写 | NFS 共享存储 |

### 8. PVC RWO，PV RWX？

**能**。PVC 所需 modes 是 PV 的子集即可。

### 9. 绑定条件

容量够、accessModes 匹配、**storageClassName 一致**、selector 匹配、PV 状态 Available。

### 10. Pending 原因

无匹配 PV；SC 不一致；容量/模式不对；动态供给失败（Provisioner/配额）；所有 PV 已 Bound。

### 11. SC 不一致

绑定控制器要求 PVC 与 PV 的 `storageClassName` **相同**（或都未指定），否则不匹配。

### 12. 容量

- PVC 5Gi + PV 10Gi → **可以**绑。  
- PVC 10Gi + PV 5Gi → **不可以**。

## 三、生命周期与回收

### 13. 五阶段

Provisioning → Binding → Using → Releasing → Reclaiming。

### 14. Pod 删了？

**Pod 删了**：PVC **通常还在**（除非被删）。  
**数据**：在 PV 底层卷上，直到 PVC 删除触发回收。

### 15. 回收策略

| 策略 | 行为 | 生产库 |
|------|------|--------|
| **Retain** | 删 PVC 后 PV Released，**数据保留** | **推荐**重要数据 |
| **Delete** | 删 PVC 后删 PV，**底层卷常删** | 临时/动态云盘 |
| Recycle | 已废弃 | 不用 |

### 16. Released

PVC 已删，PV 未回收完；**不会自动**绑新 PVC，需管理员处理。

### 17. Recycle

已废弃；推荐 **动态供给 + Delete** 或重要数据 **Retain**。

### 18. 动态云盘删 PVC

`reclaimPolicy: Delete` 时，PVC 删除 → PV 删除 → **云盘常一并删除**。

## 四、实战与进阶

### 19. Pod 挂 PVC；Deploy 多副本 + RWO

```yaml
volumes:
- persistentVolumeClaim:
    claimName: my-pvc
```

**RWO** 只能挂 **一个节点**：Deployment 多副本若都挂同一 RWO PVC，只有一份能挂载成功。  
多副本要：**每 Pod 独立 PVC**（STS + volumeClaimTemplates）或 **RWX**。

### 20. volumeClaimTemplates

STS 为 **每个 Pod 序号** 自动创建一个 PVC（如 `data-mysql-0`），实现 **每副本独立盘**。

### 21. 职责分工

见笔记 §7 表：管理员 PV/SC；开发者 PVC + 挂载。

### 22. CSI

**Container Storage Interface**：统一存储插件接口，新集群多用 CSI 驱动替代旧 in-tree provisioner。

### 23. allowVolumeExpansion

StorageClass 开启后，可对 **Bound 的 PVC** 扩容（需存储后端支持）。

### 24. Local PV

使用节点本地盘；Pod 需 **调度到该节点**（nodeAffinity），节点挂了数据可能不可用。

### 25. 排查命令

```bash
kubectl get sc,pv,pvc -A
kubectl describe pvc <name> -n <ns>
kubectl describe pv <name>
kubectl get pods -o wide -n <ns>   # 看挂卷节点
```

## 五、陷阱

| # | 简答 |
|---|------|
| F1 | **RWO 不能**多 Pod 同时读写；RWX 才可以（如 NFS） |
| F2 | 无 SC 的 PVC 只能绑 **无 SC 的 PV**；有默认 SC 时行为看集群配置 |
| F3 | **PV 存数据/库文件**；ConfigMap/Secret 适合 **配置** 小文件，不替代数据库盘 |

## 六、现网

### 26. Pulsar / MySQL

- **MySQL STS**：`volumeClaimTemplates` → 每实例 PVC，通常 **RWO**。  
- **Pulsar/Bookie**：常见多 PVC 存日志，**describe sts** 确认。

### 27

```bash
kubectl get storageclass
kubectl get pvc -A | findstr Bound
```

### 28

生产 MySQL/Pulsar 数据：**倾向 Retain** 防误删；临时卷可用 Delete。
