## 一、Service

### 1. 为什么需要 Service？

Pod 重建/扩缩容后 **IP 变化**；Service 通过 **Label Selector + 稳定 DNS/ClusterIP** 提供固定入口。

### 2. 谁转发？

**Service 不转发**；每节点 **kube-proxy** 根据 Service/Endpoints 写 **iptables 或 IPVS** 规则完成转发。

### 3. 四种类型

| 类型 | 场景 |
|------|------|
| ClusterIP | 集群内默认 |
| NodePort | 节点 IP + 高位端口 |
| LoadBalancer | 对外 VIP（云 LB 或 MetalLB） |
| ExternalName | 集群内 DNS 指到外部域名 |

### 4. Headless

`clusterIP: None`；DNS 返回 **各 Pod A 记录**，不做 VIP 负载。用于 **StatefulSet** 固定身份访问。

### 5. 负载均衡链路

```text
DNS → Service VIP → kube-proxy 规则 → Endpoints 中某 Pod IP → CNI 到 Pod
```

kube-proxy 监听 Endpoints 变化并更新规则；默认轮询等策略视模式而定。

### 6. iptables vs IPVS

| | iptables | IPVS |
|---|----------|------|
| 机制 | 规则链 | 哈希表 |
| 规模 | 规则多变慢 | 大规模更合适 |
| 备注 | 成熟 | 可能要 strictARP（+ MetalLB L2，见 D2） |

### 7. Endpoints

Service 后端 Pod IP 列表；**空** = 无 Ready Pod 或 selector 错 → VIP 无后端。

### 8. ExternalName

把集群内服务名 **CNAME** 到外部域名，不分配 ClusterIP，用于访问集群外服务。

### 9. 为何不直接用 Pod IP

不固定；Service 是稳定抽象。

### 10. ClusterIP 在网卡上吗？

**否**；虚拟 IP，由 kube-proxy 拦截处理。

### 11. LB 为何有 NodePort

K8s 默认：LoadBalancer **包含** NodePort + ClusterIP；生产对外用 **EXTERNAL-IP**，NodePort 多用于调试。

### 12. 集群内外访问

| 访问方 | 通常走 |
|--------|--------|
| 集群内 Pod | **ClusterIP** 或 `svc.ns.svc` DNS |
| 集群外/设备 | **LoadBalancer VIP**（你们 `.190`） |

## 三、Ingress

### 16. vs Service

| | Service | Ingress |
|---|---------|---------|
| 层 | **L4** | **L7** HTTP(S) |
| 能力 | IP+端口 | Host、Path、TLS |

### 17. 为何要 Controller

Ingress 资源只是 **规则**；**Controller**（如 ingress-nginx）监听 API 并配置真实代理。

### 18. 如何感知变化

Controller **watch** Ingress、Service、Secret 等，动态 reload 配置。

### 19. MQTT 与 Ingress

标准 Ingress 面向 **HTTP**；**1883 MQTT** 常用 **LoadBalancer/TCP Service**。你们用 VIP:1883 合理。

### 20. 多服务

一个 Ingress 多条 `rules`：不同 `host` 或 `path` 指向不同 **Service** backend。

## 四、NetworkPolicy

### 21. 默认

**全开放**，所有 Pod 可互访（除非 CNI/策略另行限制）。

### 22. 白名单

对被选中的 Pod：通常只 **允许** 规则中声明的流量；未匹配可 deny（视实现与默认策略）。

### 23. Ingress / Egress

| 词 | 含义 |
|----|------|
| NP **Ingress** | 控制 **进入** 该 Pod 的流量 |
| NP **Egress** | 控制 **从** 该 Pod **发出** 的流量 |

（勿与 K8s Ingress 资源混淆。）

### 24. Flannel

Flannel 只做 **Overlay 连通**，无策略引擎；NP 需 **Calico/Cilium** 等。

## 五、CNI

### 25. CNI 职责

分配 Pod IP、配置 netns、跨节点连通；**不管** Service VIP（那是 kube-proxy）。

### 26. 跨主机两种实现

- **Overlay**（VxLAN 等）：封装隧道。  
- **路由**（Calico BGP 等）：节点间路由 Pod 网段。

### 27. VxLAN vs BGP

| | VxLAN | BGP |
|---|-------|-----|
| 配置 | 简单 | 要网络配合 |
| 性能 | 有封装开销 | 通常更好 |
| 依赖 | 三层可达即可 | 路由器/交换机 BGP |

### 28. Flannel vs Calico

见笔记 §6.2 表：Flannel 简单无 NP；Calico 生产常用、支持 NP 与 BGP。

### 29. Calico BGP vs MetalLB BGP

**不是**。Calico 宣告 **Pod CIDR**；MetalLB 宣告 **Service VIP**。

### 30. 如何选

测试/学习 → Flannel；生产要策略 → **Calico/Cilium**；极致性能 → Cilium eBPF。

### 31. 同 Pod / 跨 Pod

同 Pod 内容器：**loopback**；跨 Pod：veth + CNI 网络。

### 32. Canal

**Flannel** 提供网络 + **Calico** 提供 NetworkPolicy 组件的组合方案。

## 六、全链路

### 33. HTTP + Ingress

```text
外网 → DNS → Ingress Controller (L7)
     → Service ClusterIP (L4) → kube-proxy → Pod IP (CNI)
```

### 34. 你们 MQTT/HTTP

```text
设备 → 192.168.27.190:1883/80
     → MetalLB L2 (node6) → kube-proxy → iot-web Endpoints → Pod
```

### 35. VIP 通但失败

分层查：**Endpoints** → Pod Ready → 应用端口 → NetworkPolicy → 节点防火墙。

## 七、陷阱

| # | 简答 |
|---|------|
| F1 | ClusterIP 类型不会有 EXTERNAL-IP；要 LB 类型 |
| F2 | 无 Controller，Ingress 规则 **不生效** |
| F3 | **不能**；NP 管允许/拒绝，不做服务发现 |
| F4 | CNI 坏则 Pod 网络不通，Service 转发也到不了 Pod |
